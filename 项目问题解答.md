## 1. Mock 和 Real 的区别

### 1.1 切换机制

模式切换由环境变量 `NEXT_PUBLIC_API_MODE` 控制，在 `frontend/src/lib/api/index.ts` 中实现：

```typescript
const mode = process.env.NEXT_PUBLIC_API_MODE || 'mock';
export const api = mode === 'real' ? realApi : mockApi;
```

默认值为 `mock`，即不设置环境变量时使用 Mock 模式。

### 1.2 Mock 模式（`frontend/src/lib/api/mock.ts`）

| 特征 | 说明 |
|---|---|
| **数据来源** | 读取 `contract/samples/*.json` 静态样本文件（flow、alert、graph、investigation、recommendation 等共 10 个） |
| **存储方式** | 内存中的 JavaScript 数组（`mockPcaps`、`mockFlows`、`mockAlerts` 等），页面刷新后数据重置 |
| **告警变体** | 硬编码 5 种告警类型变体：`bruteforce`、`port_scan`、`data_exfiltration`、`lateral_movement`、`c2_beacon`，模拟不同安全场景 |
| **PCAP 处理** | 模拟异步处理：用 `setTimeout` 模拟进度 10%→40%→70%→100%，随机生成 flows 和 alerts，无真实解析 |
| **WebSocket** | 通过 `wsClient.simulateEvent()` 在前端本地触发事件，不需要后端 WS 服务 |
| **网络请求** | **不发送任何 HTTP 请求**，所有 API 调用直接操作内存数据并返回 |
| **后端依赖** | 完全不需要后端运行 |
| **适用场景** | 前端开发调试、UI 演示、无后端环境下独立测试 |

### 1.3 Real 模式（`frontend/src/lib/api/real.ts`）

| 特征 | 说明 |
|---|---|
| **数据来源** | 调用后端 REST API（`http://localhost:8000/api/v1/...`） |
| **通信协议** | HTTP + JSON，所有响应遵循信封格式 `{ok, data, error}`，由 `fetchJson<T>()` 自动解包 |
| **PCAP 处理** | 真实上传文件到后端，由后端 `_process_pcap_sync()` 执行完整管线（解析→特征→检测→告警） |
| **WebSocket** | 连接真实的后端 WS 端点 `ws://localhost:8000/api/v1/stream/ws`，接收实时进度和告警推送 |
| **错误处理** | 有完整的 HTTP 状态码检查和 envelope 错误提取 |
| **后端依赖** | **必须启动后端服务** |
| **适用场景** | 真实部署、集成测试、完整功能验证 |

### 1.4 应该用哪种模式？

- **开发/演示阶段**：使用 `mock` 模式，前端可独立运行，无需后端
- **正式部署/测试阶段**：使用 `real` 模式，连接真实后端
- **不建议混用**：`api` 对象是全局单例，应用运行时只能是一种模式

### 1.5 Mock 相关文件需要删除吗？

**不建议删除。** 理由如下：

1. Mock 文件仅 `mock.ts` 一个（496 行），体积很小，不影响生产打包（tree-shaking 会排除未引用的模块）
2. `contract/samples/*.json` 同时作为 API 契约文档的参考数据，有独立价值
3. 保留 Mock 模式便于后续前端独立开发、快速演示和单元测试
4. 如果确实想删除，只需删除 `frontend/src/lib/api/mock.ts` 并修改 `index.ts` 直接导出 `realApi` 即可

## 2. 切换到 Real 模式需要修改什么

### 2.1 最小改动清单

| 步骤 | 文件 | 操作 |
|---|---|---|

| **② 设置后端地址** | `frontend/.env.local` | 添加 `NEXT_PUBLIC_API_BASE_URL=http://localhost:8000`（默认值即此，如后端地址不同则需修改） |
| **③ 启动后端** | 终端 | 在 `backend/` 目录下运行 `uvicorn app.main:app --host 0.0.0.0 --port 8000` |
| **④ 启动前端** | 终端 | 在 `frontend/` 目录下运行 `npm run dev` |

### 2.2 环境变量文件示例

在 `frontend/` 目录下创建 `.env.local`：

```env
NEXT_PUBLIC_API_MODE=real
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000
```

> **注意**：Next.js 的 `NEXT_PUBLIC_` 前缀变量会被编译到客户端 JS 中，修改后需要**重启前端开发服务器**才生效。

### 2.3 你需要提供/准备的东西

1. **Python 环境**：Python 3.11+，安装 `backend/requirements.txt` 中的依赖
2. **PCAP 文件**：用于上传分析的网络抓包文件（项目 `backend/data/` 中已有示例）
3. **磁盘空间**：SQLite 数据库会自动创建在 `backend/data/nettwin.db`，PCAP 文件存储在 `backend/data/pcaps/`
4. **无需额外数据库**：项目使用 SQLite，`init_db()` 在 FastAPI 启动时自动建表

### 2.4 Docker 方式

如果使用 Docker，在 `infra/docker-compose.yml` 中，前端服务的 `environment` 部分设置：

```yaml
environment:
  - NEXT_PUBLIC_API_MODE=real
  - NEXT_PUBLIC_API_BASE_URL=http://backend:8000
```

## 3. Real 模式下 Agent 是否针对不同 PCAP 进行针对性分析

**是的，每个 PCAP 都是独立分析的，不同 PCAP 会产生不同的、针对性的告警和分析结果。**

### 3.1 PCAP 独立处理

每个上传的 PCAP 由 `_process_pcap_sync()` 函数独立处理（`backend/app/api/routers/pcaps.py`）：

```
PCAP_A.pcap  →  解析出 flows_A  →  特征提取_A  →  异常评分_A  →  告警_A
PCAP_B.pcap  →  解析出 flows_B  →  特征提取_B  →  异常评分_B  →  告警_B
```

关键机制：
- **IsolationForest 是 per-PCAP 训练的**：`DetectionService` 每次调用 `score_flows()` 时重新 `fit(X)` 模型。这意味着异常评分是**相对于当前 PCAP 的流量基线**计算的
- **告警聚合是 per-PCAP 的**：`AlertingService.generate_alerts(flows, pcap_id)` 只对当前 PCAP 的异常流量进行聚合

### 3.2 告警类型是针对性识别的

`AlertingService._determine_type()` 方法根据流量模式自动判定告警类型：

| 告警类型 | 判定规则 |
|---|---|
| `scan`（扫描） | 目标 IP > 5 个 **或** 目标端口 > 10 个 **或** SYN ratio > 50% |
| `bruteforce`（暴力破解） | 单一目标 + 连接数 > 5 + 目标端口为 22/23/3389/21 |
| `dos`（拒绝服务） | 总字节 > 1MB + 单一目标 IP |
| `anomaly`（通用异常） | 不匹配以上任何模式的默认类型 |

### 3.3 Agent 分析也是针对性的

`AgentService` 的三个方法（`triage`、`investigate`、`recommend`）都接收具体的 `Alert` 对象，基于告警的类型、严重性、源/目标 IP、协议等属性生成针对性内容：

- **Triage（分诊）**：根据 alert.type 生成不同描述（如 scan vs bruteforce 的描述完全不同）
- **Investigate（研判）**：`_build_hypothesis()` 为每种告警类型构建不同假设
- **Recommend（建议）**：`_build_actions()` 为 scan/bruteforce 推荐封禁 IP，为 dos 推荐速率限制

### 3.4 示例对比

| 维度 | 扫描攻击 PCAP | 暴力破解 PCAP |
|---|---|---|
| 流量特征 | 多目标 IP、多目标端口、高 SYN ratio | 单目标、端口 22、高连接频率 |
| 异常评分 | 基于该 PCAP 内的流量分布 | 基于该 PCAP 内的流量分布 |
| 告警类型 | `scan` | `bruteforce` |
| Agent 假设 | "Possible port scanning from X" | "Possible brute-force attack targeting SSH" |
| 推荐动作 | 封禁源 IP + 加强监控 | 封禁源 IP + 检查防火墙 |

## 4. 项目的作用

### 4.1 一句话概括

**NetTwin SOC** 是一个**网络安全数字孪生平台**，通过分析 PCAP 网络抓包文件，自动检测异常流量、生成安全告警、辅助安全研判，并在数字孪生拓扑上模拟防御动作的效果。

### 4.2 核心功能

```
上传 PCAP → 解析流量 → 提取特征 → AI 异常检测 → 生成告警
                                                       ↓
                              ← 推演模拟 ← 动作规划 ← Agent 研判
```

1. **PCAP 上传与解析**：支持 `.pcap` / `.pcapng` 格式的网络抓包文件上传和自动解析
2. **流量特征提取**：从原始网络包中提取 22 维特征向量（包大小、持续时间、IAT、TCP 标志位等）
3. **AI 异常检测**：使用 IsolationForest 机器学习模型对每条流量评分，识别异常行为
4. **智能告警生成**：将异常流量按规则聚合为告警，自动分类（扫描/暴力破解/DoS/异常/数据外泄）
5. **Agent 辅助研判**：自动生成分诊摘要、调查假设、影响评估和应对建议
6. **数字孪生拓扑**：基于流量数据构建 3D 网络拓扑图，支持 IP/子网视图和时间滑块
7. **Dry-Run 推演**：在数字孪生拓扑上模拟封禁/隔离/限速等动作，评估对网络连通性的影响
8. **场景回归测试**：预定义安全场景（如已知的扫描攻击 PCAP），验证检测引擎是否正确产生预期告警

### 4.3 技术栈

| 层 | 技术 |
|---|---|
| 前端 | Next.js (App Router)、TypeScript、React Flow (2D 证据链)、react-three-fiber (3D 拓扑)、Tailwind CSS |
| 后端 | Python 3.11、FastAPI、SQLAlchemy 2.0、Pydantic v2 |
| 检测 | dpkt（PCAP 解析）、scikit-learn IsolationForest（异常检测） |
| 数据库 | SQLite（自动创建，11 张表） |
| 部署 | Docker Compose |

## 5. PCAP 是怎么到 Alert 的

### 5.1 完整管线（7 步）

以下是 `backend/app/api/routers/pcaps.py` 中 `_process_pcap_sync()` 的完整流程：

```
┌──────────────────────────────────────────────────────────────────┐
│                    POST /api/v1/pcaps/{id}/process               │
│                    mode = "flows_and_detect"                     │
└──────────────────────────────┬───────────────────────────────────┘
                               │
                               ▼
┌─ Step 1 ─────────────────────────────────────────────────────────┐
│  更新状态 → processing, progress = 10%                           │
│  WebSocket 广播: pcap.process.progress {percent: 10}             │
└──────────────────────────────┬───────────────────────────────────┘
                               ▼
┌─ Step 2: 解析 PCAP ─────────────────────────────────────────────┐
│  ParsingService.parse_to_flows(pcap_file, window_sec)            │
│  - 使用 dpkt 逐包读取 PCAP 文件                                  │
│  - 按五元组 + 时间窗口聚合为 Flow（流量记录）                       │
│  - 提取 TCP 标志位、包大小、时间戳等原始信息                        │
│  输出: flow_dicts[]    progress = 40%                            │
└──────────────────────────────┬───────────────────────────────────┘
                               ▼
┌─ Step 3: 特征提取 ──────────────────────────────────────────────┐
│  FeaturesService.extract_features_batch(flow_dicts)              │
│  - 为每条 Flow 计算 22 维特征:                                    │
│    total_packets, total_bytes, bytes_per_packet, flow_duration_ms │
│    fwd_ratio_packets, fwd_ratio_bytes                            │
│    iat_mean_ms, iat_std_ms (包间到达时间)                         │
│    syn/ack/fin/rst/psh_count (TCP 标志位)                        │
│    syn_ratio, rst_ratio                                          │
│    dst_port_bucket, is_tcp/udp/icmp                              │
│    avg_pkt_size_fwd, avg_pkt_size_bwd                            │
│  输出: flow_dicts[] (每条附带 features 字段)  progress = 55%     │
└──────────────────────────────┬───────────────────────────────────┘
                               ▼
┌─ Step 4: 异常检测 ──────────────────────────────────────────────┐
│  DetectionService.score_flows(flow_dicts)                        │
│  - 提取 18 个数值特征构建特征矩阵 X                               │
│  - 训练 IsolationForest 模型:                                    │
│    contamination=0.1, n_estimators=100, random_state=42          │
│  - 计算每条 Flow 的 anomaly_score (0~1, 越高越异常)               │
│  - Min-Max 归一化，取反（sklearn 原始分数越低越异常）               │
│  输出: flow_dicts[] (每条附带 anomaly_score)  progress = 70%     │
└──────────────────────────────┬───────────────────────────────────┘
                               ▼
┌─ Step 5: 写入数据库 ────────────────────────────────────────────┐
│  将所有 flow_dicts 写入 flows 表                                  │
│  progress = 85%                                                  │
└──────────────────────────────┬───────────────────────────────────┘
                               ▼
┌─ Step 6: 生成告警 ──────────────────────────────────────────────┐
│  AlertingService.generate_alerts(flow_dicts, pcap_id)            │
│  ① 过滤: anomaly_score >= 0.7 的流量                             │
│  ② 聚合: 按 src_ip + 时间窗口(window_sec) 分组                   │
│  ③ 类型判定: scan / bruteforce / dos / anomaly                   │
│  ④ 严重性: critical ≥ 0.95 / high ≥ 0.85 / medium ≥ 0.70       │
│  ⑤ 构建 evidence: {flow_ids, top_flows, top_features, pcap_ref} │
│  ⑥ 写入 alerts 表 + alert_flows 关联表                           │
│  ⑦ WebSocket 广播: alert.created {alert_id, severity}            │
│  progress = 95%                                                  │
└──────────────────────────────┬───────────────────────────────────┘
                               ▼
┌─ Step 7: 完成 ──────────────────────────────────────────────────┐
│  pcap.status = "done", progress = 100%                           │
│  WebSocket 广播: pcap.process.done {flow_count, alert_count}     │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 关键参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `mode` | `flows_and_detect` | `flows_only` 只解析不检测，`flows_and_detect` 完整管线 |
| `window_sec` | `60` | 流量聚合时间窗口（秒），同一源 IP 在窗口内的异常流量聚合为一条告警 |
| `score_threshold` | `0.7` | 异常分数阈值，高于此值的流量被视为异常 |
| `contamination` | `0.1` | IsolationForest 参数，假设 10% 的流量为异常 |

## 6. 项目整体中英文切换实现

### 6.1 当前状态：**未实现国际化**

经过对整个项目的全面分析，**前端目前完全没有国际化（i18n）机制**：

| 检查项 | 结果 |
|---|---|
| `package.json` 中的 i18n 库 | **无** — 没有 `next-intl`、`react-intl`、`i18next` 等任何国际化依赖 |
| locale 配置文件 | **无** — 不存在 `messages/`、`locales/`、`i18n.ts` 等 |
| 语言切换 UI | **无** — 没有语言选择下拉框或切换按钮 |
| `<html lang>` 属性 | 硬编码 `"en"`（`layout.tsx`） |
| 所有页面文本 | **英文硬编码字符串**（标题、描述、按钮、toast 消息等） |

### 6.2 后端仅有的多语言支持

后端 Agent Service 的 `triage()` 方法支持 `language` 参数（`"zh"` 或 `"en"`）：

```python
# backend/app/services/agent/service.py
def triage(self, alert: Alert, language: str = "en") -> str:
    if language == "zh":
        summary = f"检测到{alert.severity}级{self._type_to_zh(alert.type)}异常..."
    else:
        summary = f"Detected {alert.severity} severity {alert.type} anomaly..."
```

但前端 `AgentPanel.tsx` 中调用时硬编码为 `'en'`：

```typescript
const triageData = await api.triage(alertId, { language: 'en' });
```

### 6.3 如何实现中英文切换（推荐方案）

如果要为项目添加完整的国际化支持，推荐使用 **`next-intl`** 库（与 Next.js App Router 集成最好）：

**步骤概述：**

1. 安装 `next-intl`：`npm install next-intl`
2. 创建 locale 文件：`frontend/messages/en.json` 和 `frontend/messages/zh.json`
3. 配置 `next.config.ts` 添加 i18n 设置
4. 在 `layout.tsx` 中添加 `NextIntlClientProvider`
5. 将所有硬编码字符串替换为 `useTranslations()` 调用
6. 添加语言切换组件（如在 `TopBar` 中添加下拉菜单）
7. 修改前端 Agent API 调用，将 `language` 参数与用户选择的语言联动

**改动范围估计：**
- 7 个页面文件
- ~15 个组件文件
- 1 个布局文件
- 约 200+ 个硬编码文本需要提取为翻译 key

## 7. Agent 如何实现、能否升级、是否需要大模型 API

### 7.1 当前实现：规则/模板驱动

`AgentService`（`backend/app/services/agent/service.py`，318 行）使用**纯规则和字符串模板**实现，**不调用任何 LLM API**。

三个核心方法：

| 方法 | 功能 | 实现方式 |
|---|---|---|
| `triage(alert, language)` | 生成分诊摘要 | 字符串模板：基于 alert 的 type、severity、src_ip、flow_count 等字段拼接 |
| `investigate(alert)` | 生成调查报告 | 调用 4 个子方法：`_build_hypothesis()`（假设）、`_build_why()`（原因）、`_assess_impact()`（影响）、`_suggest_next_steps()`（下一步） |
| `recommend(alert)` | 生成行动建议 | `_build_actions()` 根据 alert.type 生成不同的行动列表，每个行动包含 title、steps、rollback、risk |

**举例 — `_build_hypothesis()`：**

```python
hypotheses = {
    "scan": f"Possible port scanning or network reconnaissance from {alert.primary_src_ip}",
    "bruteforce": f"Possible brute-force attack targeting {alert.primary_proto}/{alert.primary_dst_port}",
    "dos": f"Possible denial-of-service attack against {alert.primary_dst_ip}",
    ...
}
return hypotheses.get(alert.type, hypotheses["anomaly"])
```

这是一个纯 if-else 映射，没有任何模型推理。

### 7.2 能否升级？

**完全可以，且架构已经预留了升级空间。** `AgentService` 是一个独立的 Service 类，修改其内部实现不影响其他组件。

### 7.3 需不需要大模型 API？

**当前不需要，但升级后需要。**

| 方案 | 大模型 API | 适用场景 |
|---|---|---|
| 当前方案（模板） | ❌ 不需要 | 演示、教学、不想依赖外部服务 |
| 升级方案 A（API 调用） | ✅ 需要 OpenAI / Claude / 通义千问等 API Key | 生产环境，需要更智能的分析 |
| 升级方案 B（本地模型） | ❌ 不需要外部 API | 对数据安全有要求，可部署 Ollama/vLLM 本地推理 |

### 7.4 需不需要针对性训练？

**不需要微调/训练**。原因：

1. **Prompt Engineering 即可**：安全分析是 LLM 通用能力的一部分，通过精心设计的 System Prompt +　Few-shot 示例即可获得高质量输出
2. **结构化输出**：可以使用 OpenAI 的 Function Calling 或 JSON Mode，让 LLM 返回与当前 `InvestigationSchema` / `RecommendationSchema` 完全兼容的结构化 JSON
3. **RAG 增强**：如果有企业特定的安全知识库（如已知威胁情报、内部网络策略），可以用 RAG 注入上下文，而不需要微调

## 9. 特征→流量→告警→研判→动作→推演的完整流程与 AI 介入点

### 9.1 完整流程图

```
用户上传 PCAP
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ ① 流量解析 (ParsingService)                                  │
│    PCAP 原始数据包 → Flow 流量记录                             │
│    dpkt 逐包读取，按五元组+时间窗口聚合                         │
│    ⚡ AI 介入: 无                                             │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐

│    ⚡ AI 介入: 无（纯统计计算）                                │
└──────────────────────────────┬──────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ ③ 异常检测 (DetectionService)                                 │

│    异常流量 → Alert 告警                                      │
```

### 9.2 AI 介入点汇总

| 步骤 | AI 技术 | 当前状态 | 可升级方向 |
|---|---|---|---|
| ① 流量解析 | — | 无 AI | 无需 AI |
| ② 特征提取 | — | 无 AI | 可用 AutoEncoder 自动学习特征 |
| ③ **异常检测** | **IsolationForest** | **✅ 已实现** | 可升级为 Deep Learning（AutoEncoder/LSTM） |
| ④ 告警生成 | — | 规则引擎 | 可用 ML 进行更智能的聚合和分类 |
| ⑤ **Agent 研判** | 模板规则 | **⚠️ 可升级** | **接入 LLM API（GPT-4o/Claude/通义千问）** |
| ⑥ 动作规划 | — | 用户+模板 | LLM 可自动生成动作方案 |
| ⑦ 推演模拟 | — | 图算法 | 可用 GNN 预测网络影响 |

### 9.3 关键结论

- **当前项目中实际使用 AI 的环节只有一个**：Step ③ 的 IsolationForest 异常检测
- **最值得升级的环节**：Step ⑤ Agent 研判 — 接入 LLM 后可大幅提升分析质量
- **不需要训练/微调**：IsolationForest 是无监督模型，每次自动 fit；LLM 通过 Prompt Engineering 即可
- **整体架构是 "ML 检测 + 规则引擎 + 图算法" 的组合**，LLM 作为后续升级方向
